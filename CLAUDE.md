# SPIN CLASS — Project Instructions

SPIN CLASS is Dave's fitness web app for himself and friends. Joke name — there are no bikes; the app owns this with dry humor. Bold dark theme: bg #0b0b0f, electric lime accent #ccff00, pink-red #ff3d71, cyan #00e0ff. Tone: energetic gym-buddy, minimal emoji.

## Stack & repo
- GitHub: creativecowboys/spin-class → Vercel project "spin-class" (team Creative Cowboys), auto-deploys on commit to main
- No build step, no framework. Files:
  - `index.html` — the entire app (single file: HTML+CSS+JS, ~80KB). Embedded: 366-exercise library (EXDB const, from Dave's "Gym Exercise Database.xlsx") and the workout program (PROGRAM, seeded as PPL Strength A/B; NO LONGER fixed at 7 — it's a variable-length library of 1–60 workouts). A workout counts as a rest day by NAME (isRestDay = /rest/i.test(name)), not a hardcoded index. A schedule day can hold MULTIPLE workouts: `S.schedule[iso]` is a single ref (`PROGRAM index` number or `"c:<id>"` custom) OR an **array** of refs — always read via `dayRefs(iso)` / write via `setDayRefs(iso, refs)` (a single ref is stored as a scalar for back-compat, several as an array). The day-assign picker (openDayAssign) shows each workout as a COLLAPSIBLE preview card (tap the header → togglePreview expands its exercise list; tap the `+`/`✓` circle → toggleDay adds/removes it for the day; preview open-state in the `pickerOpen` Set survives re-render); multi-select via `toggleDay`; the Train "today" card + week cal + renderProgram all reflect all of a day's workouts. In the live workout, each set row has a ✕ (delSet) to remove that set
  - `admin.html` — admin console (member viewer/editor, Workout Library builder, per-member schedule assignment). Has its own EXDB + DEFPROG copies embedded
  - `api/chat.js` — Claude coach proxy (model claude-sonnet-5, Anthropic Messages API, coach system prompt; never gives supplement/peptide dosing advice — reminders only). When recommending a concrete food, the coach appends a hidden `[[LOG_MEAL:{"name","cal","protein","carbs","fat"}]]` tag; the app (parseCoachMeals/coachMsgHtml in index.html) strips it from the visible reply and renders a "+ Log" button that prefills the meal modal for today. Profile has a goal field (S.profile.goal = cut|maintain|bulk, segmented selector in the Edit Profile modal, shown in the summary as Cutting/Maintaining/Bulking); it's in the coach context. The coach can recommend daily targets and emit a `[[SET_TARGETS:{cal,protein,carbs,fat}]]` tag → the app renders a "✓ Set my targets" button (applyCoachTargets) that writes them into the profile dashboard. A "Dial in my targets" chat chip kicks off the setup. The coach input has a 📷 button (onCoachPhoto/sendCoachImage) to scan a nutrition label: the compressed photo is sent as an image content block in the Messages API (sonnet-5 vision), the coach reads the macros and emits a LOG_MEAL tag → one-tap log. The label image is NOT persisted in S.chat (a text placeholder is stored) to keep synced state small
  - `api/sync.js` — device-state sync (GET ?id= / POST {id, state}) plus name+PIN account lookup (GET ?name=&pin= → matches list; accounts with no S.pin are claimable by name alone; 403 wrong pin / 404 no match). Device IDs are random 24-hex secrets generated at onboarding
  - `api/community.js` — team feed + crew chat (GET ?id= requires a valid member id; returns per-member name/**group**/streak/recent-workout summaries + the caller's-crew chat log — never weights, meals, stack, coach chats, PINs, or device ids; members are identified by a sha256-derived 12-char key). **Chat is per-crew**: keyed off the caller's `state.profile.group`, stored in `config/teamchat-<slug>.json` (slug = lowercased group name, non-alnum→`-`); ungrouped members share the legacy `config/teamchat.json`. Still NOT in member state (avoids last-write-wins clobber; sender name resolved server-side; last 100 kept). POST {id, text} appends to the caller's crew chat; POST {id, delAt} deletes one of the caller's own messages. (Feed ROSTER is still returned in full and filtered client-side in index.html by `S.profile.group`.)
  - `api/photo.js` — progress-photo storage (POST {id, dataUrl} → resizes client-side first, stores JPEG in blob at `photos/<id>/…` with addRandomSuffix for unguessable URLs, returns {url}; DELETE {id, url} only within the caller's own photo path). App keeps just `S.photos=[{url,date,at}]` so synced state stays small; photos are private (not in community feed or admin)
  - `api/users.js`, `api/admin-user.js` (PUT/DELETE), `api/program.js` (GET public, PUT admin — accepts a `days` array of 1–60 workouts, each needing a name + ex array) — admin endpoints check `x-admin-key` header against ADMIN_KEY env
  - `package.json` (@vercel/blob, type:module), `vercel.json` (cleanUrls)
- Storage: Vercel Blob store "spin-class-blob" (public access; paths `users/<id>.json`, `config/program.json`; cacheControlMaxAge 0)
- Env vars on Vercel: ANTHROPIC_API_KEY (Dave's platform.claude.com account — separate billing from his Claude sub, spend cap set), ADMIN_KEY, BLOB_READ_WRITE_TOKEN + BLOB_STORE_ID (from store connection)

## App architecture notes
- **Crews (groups):** a member's crew is the free-text `S.profile.group` (e.g. "Ryze Guys", "Dixon's"; blank = ungrouped). Admin assigns/edits it (members list is sorted+grouped by it; per-member Group input with a `#grp-list` datalist of crew names). Crews are socially separate: the Team feed roster is filtered client-side (`effectiveMembers`, by `S.profile.group`) and **chat is separate per crew server-side** (see api/community.js). Workouts live in ONE shared library but each workout can be tagged `group: "<crew name>"` (blank = all crews); `workoutVisible(p)` (index.html) = `!p.group || p.group===S.profile.group`, gating the Train tab list + day-assign picker so members only see their crew's (+ untagged) workouts. Tags only affect visibility, not resolution — schedule refs are raw PROGRAM indices, so an already-scheduled workout always resolves. Admin: per-workout **Crew** tag + crew filter chips on the library; per-member schedule dropdowns show only that member's crew's workouts; **Crew Schedule** card pushes a week to one crew (Everyone or a specific crew, only that crew's workouts offered). Keep the group model in sync: index.html `workoutVisible`/`effectiveMembers`, admin.html `knownCrews`/`workoutVisibleToCrew`/`crewMembers`, api/community.js `groupOf`/`chatPath`.
- All state in one object `S`, persisted to localStorage key "forge" (legacy name), synced to server: debounced POST on every save + sync on open. On load, pulls master program and replaces whole local state if server copy is >2s newer (last-write-wins; admin pushes rely on this). loadRemote preserves an in-progress workout (S.active) ACROSS the state pull — so an admin schedule push lands even mid-workout (data updates, the live workout screen stays; it shows on Finish). The visibilitychange re-pull is no longer gated by !S.active (the pull is now safe during a workout)
- **Save robustness:** every save/delete calls `closeModal()` BEFORE `renderAll()` — if a render ever throws, the modal still closes, so you can't accidentally double-submit (this was double-logging meals). `store.save()` sets `S.updatedAt=Date.now()` synchronously so a return-to-foreground pull can't clobber a just-saved entry.
- **Stale in-progress workout:** an `S.active` workout started on an EARLIER calendar day (`activeStale()`) no longer hijacks the Train tab — it shows a "Workout in progress" resume banner (Resume / Finish now / Discard) at the top of the normal Train view, so Recent Sessions stay visible (a forgotten active workout was hiding them). finishWorkout caps the logged duration at 180 min so a days-old session doesn't record a multi-day time.
- **Progress Photos** auto-expand on first render when any exist (photosAutoExpanded flag) and after adding one (savePhoto → expandColl('photos') + scrollIntoView) — Profile isn't a nav tab, so a collapsed section made added photos hard to find.
- Home hero shows the greeting + latest crew-chat message (taps to Team); daily cal/protein progress live in the "Today's Targets" card, streak lives in the header name chip ("Dave · 3🔥"). Crew chat polls every 4s while the Chat view is open and every 20s in the background; an unread dot shows on the Team nav icon when there are new messages (tracked via localStorage `chatSeen`, not synced state).
- Profile tab layout: summary hero (Edit Profile modal = name/targets/PIN only) → collapsible Progress Photos (collapsed default) → Stack → collapsible Weight Trend at the bottom (chart + Week/Month/Year/All + Log Weight + weigh-in history; renderWeightPanel now runs from renderProfileCard, targeting on-page elements not the modal). toggleColl(k) drives the ▸/▾ collapsible sections.
- Bottom nav has 5 tabs: Home, Train, Meals, Coach, Team (Team far right). Meals is its own tab again. Profile is NOT in the nav — reached via the header name chip (top-right, "Dave · N🔥") or the home dose alert; the Profile tab holds the profile summary + Stack section. Name/targets/PIN editing + the weight-trend chart live in the profile modal ("Edit Profile" button / `openModal('profile')`)
- Features: 4-step onboarding (name, weight/goal, cal+protein targets, fresh-vs-demo); meals (own Meals tab: day navigator ‹Today›, macro summary as 4 circular progress rings at top — Cals/Protein/Carbs/Fats vs targets, color-matched to home; full CRUD add/edit/delete any meal on any date, tracks cal/protein/carbs/fats); weight trend chart (inline SVG) in the profile modal — weights are dated entries `{w, at(ms)}` (legacy `[number]` lists auto-migrate on load via migrateWeights, synthesizing one-per-day timestamps); logging takes a date + time (defaults to now); each weigh-in is editable/deletable from a tap-to-edit history list in the profile modal (so users can fix the synthesized migration dates); chart has Week/Month/Year/All views (default Week); PPL program with live workout mode (per-set weight×reps, last-weight prefill, volume totals, add/swap exercises from library); week calendar (tap a date → assign a workout; S.schedule maps ISO-date → a "workout ref" that is either a PROGRAM index (number) OR a "c:<id>" string for a user-built custom workout — resolve with workoutFor(ref)); users can Build a Workout (name + pick exercises from the library, editable sets×reps) stored in S.customWorkouts=[{id,name,sub,ex}], reusable, assignable, startable in live mode, and swipe-left-to-delete in the day picker; live workout S.active carries {ref,name,ex} not a bare day index; finished workouts reopenable/editable in place; Team area ("Team This Week" card on home with Mon–Sun training dots, full Team tab with a Chat|Crew segmented toggle — Chat is default and full-page (input pinned at bottom), Crew is the reverse-chron workout-summary feed — summary only, no set detail; members with no logged workouts still appear on the Crew roster; the current user's own row on the Team card + feed uses LOCAL workouts (effectiveMembers) so training shows instantly before the server round-trip (renderAll re-renders team views when COMM loaded); the "Log a Workout" modal has a date field for backfilling past days; Cardio logging (openCardio) — 12 cardio types + duration, reachable from the day picker ("🏃 Log Cardio") and the bottom of a live workout ("Add Cardio to This Session"), logs a `{type:'Cardio · X', cardio:true}` entry; time-based holds (isTimed(): plank/wall sit/dead hang/holds/carries) log weight + seconds instead of reps (set input shows "sec", excluded from volume); each live-workout exercise pre-fills its set fields with the prior session's weights/reps as faded placeholders (histRef/exerciseHistory from S.workouts detail; small "Faded numbers = <date>" line); a "↺ past" button (only when history exists) cycles the placeholders to older sessions (wraps); chat polls every 25s while the tab is visible; the home "Team This Week" card opens straight to the Crew view; weight trend chart lives in the profile modal, not home); peptide/supplement "Stack" scheduler (reminder-only — never suggest compounds or doses, disclaimer in UI); Claude coach chat (persists last 60 msgs in S.chat; falls back to canned coachReply() when API unavailable; chats synced but deliberately NOT shown in admin). Chat is protected from loss: after each coach message the state is flushed to the server immediately via `syncNow()` (not the 1.5s debounce), and loadRemote NEVER lets a pull shrink the conversation (keeps the longer of local/remote S.chat) — so the coach conversation carries across days/reopens for context
- Admin: members list (workout logs w/ set detail, meals, weights, stack), profile edit + delete, and a **Workout Library** builder (was "Program Builder"; renderBuilder). The library is a variable-length list of workouts, each a COLLAPSIBLE card (collapsed by default showing name + "subtitle · N exercises"; tap the header to expand/edit; open-state tracked in a WeakSet keyed by the workout object so it survives add/delete). Per-card: **Duplicate** (dupWorkout — appends a "… copy" to the END so existing day-number indices stay stable), a 🗑 **Delete** (delWorkout — blocked if it's the last one; warns that it renumbers later workouts), and the per-day exercise editor with the "+ Add from Library" picker (openExPicker: search + muscle-group chips + tap-to-add from the 366-exercise EXDB; custom free-text add still available). A top **"+ New Workout"** (newWorkout) adds a blank card. "🚀 Push to App" (pushProgram) PUTs the whole library to config/program.json (1–60, each named). Also: per-member week schedule assignment ("Push Schedule" PUTs member state) and a "Team Schedule" card (pushTeamSchedule) to assign a week to EVERYONE at once — fetches each member's freshest state first (reduces clobber), merges the schedule, PUTs each. "Fill Mon–Sun" clamps to the library length. Members reflect pushes on next app open OR on return-to-foreground (visibilitychange re-runs loadRemote; a pushed schedule now lands even mid-workout since loadRemote preserves S.active). CAVEAT: schedules store raw PROGRAM indices, so DELETING/reordering a library workout can remap members' scheduled day numbers — re-push schedules after deletes (duplicate/new append, so they're safe). NOTE: pushing the *library* only updates the exercise templates; to make a workout appear as "today's workout" it must be *scheduled* to a day (Train tab is schedule-driven — "Rest day" when nothing assigned)

## Login / account recovery (added Jul 2026)
- Name + 4-digit PIN (stored as `S.pin`, plain text in state — friends-scale trust model). New members pick a PIN at onboarding step 1; "Been here before? Log back in" on the welcome screen restores an account onto any device/URL (covers old-URL migration and cleared browser data)
- Legacy accounts (no PIN yet) can be claimed by name alone once; after restore the profile modal auto-opens to set a PIN, and a home-screen banner nudges existing members until they set one
- Multiple same-name matches → member picks by workout count + last-active date
- Admin console profile editor shows/resets each member's PIN (blank = none, re-enables claim-by-name)

## Known gaps / next up
- Onboarding now guards against duplicate accounts: finishOnboard() does a name lookup first — if the name exists with a PIN it routes to the login screen, if it exists without a PIN it reconnects (adoptAccount) instead of minting a new device id. This fixes "extra Dave keeps appearing" (root cause: a device whose localStorage was evicted re-ran onboarding and made a fresh account). Setting a PIN makes the guard airtight.
- Macro targets: profile editor has Calories/Protein/Carbs/Fats per-day targets (carbTarget/fatTarget default 250/70); the home "Today's Targets" card shows all four as colored progress bars (lime/cyan/amber/pink); coach context includes all four targets + today's totals.
- Last-write-wins sync can still clobber concurrent NON-workout edits, but an in-progress workout (S.active) is now safe — loadRemote preserves it across the pull, so an admin push mid-workout no longer wipes it (and the push still lands)
